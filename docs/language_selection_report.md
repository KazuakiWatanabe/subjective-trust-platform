# 言語選定レポート（改訂版）
## subjective-trust-platform 実装言語の技術評価

**作成日**: 2026年3月20日  
**対象リポジトリ**: KazuakiWatanabe/subjective-trust-platform  
**評価対象言語**: Python / Go / Java / C#

---

## 1. エグゼクティブサマリー

本レポートは、subjective-trust-platform の実装言語として Python・Go・Java・C# を比較評価した結果を報告する。評価は Phase 1（PoC）と Phase 2/3（本番運用・拡張）を分けて行った。

**結論：**

- **Phase 1（PoC）の第一候補は Python** とする。AI 解釈の試行錯誤速度と PoC 成功確率において優位性が明確である
- **Python が「唯一の正解」ではない**。長期運用・保守性・業務システムとしての堅牢性まで含めると、Go / Java / C# も十分に有力な選択肢である
- **最も現実的な全体構成**は、Python 単独で開始しつつ、Phase 2 以降は「業務システム本体（Go / Java / C#）+ Python AI ワーカー」の分離構成への移行を視野に入れることである

---

## 2. システムの性質と要件

### 2.1 本システムの本質

本システムは「AI システム」ではなく、**AI を一部に含む業務システム**である。この認識が言語選定の出発点として重要である。

AI が重要な構成要素であることは間違いないが、全体の大半は以下の業務領域で占められる。

- API（PWA 向け接客タグ入力・アンケート受信）
- 業務ルール（信頼スコア算出・重みテーブル・アラート判定）
- DB モデリング（TrustEvent・TrustScoreSnapshot 等 10 以上のテーブル）
- バッチ処理（日次 AI 解釈・スコア更新・週次レポート）
- 認証・運用・監査
- 将来保守（Phase 3 の Agent 層追加を含む）

AI 解釈パイプラインはこのシステムの中核的コンポーネントであるが、業務システム全体の一部である。

### 2.2 フェーズ別の優先事項

| Phase | 期間 | 優先事項 |
|---|---|---|
| Phase 1 | 3ヶ月 | PoC の速度・AI 解釈の試行錯誤・早期の店舗ダッシュボード提供 |
| Phase 2 | +3ヶ月 | 全直営店拡大・外部連携・チーム開発と保守性・障害調査容易性 |
| Phase 3 | +6ヶ月〜 | Agent 構成・Trust API・責務分離・パフォーマンス最適化 |

評価はフェーズを分けて行う。単一の加重平均で結論を出すことはしない。

## 4. 言語別評価

### 4.1 Python

#### Phase 1 適性：★★★★★

Claude API の公式 Python SDK が提供されており、AI 解釈パイプラインの試作・改善サイクルが最速である。設計書が定義する解釈出力スキーマ（`trust_dimension`・`sentiment`・`interpretation` 等の JSON）は、pydantic モデルとの親和性が高い。FastAPI により REST API も高速に立ち上がり、pandas / polars で信頼スコアの集計ロジックを簡潔に実装できる。

**クラウド適合性（GCP）**: Cloud Functions・Cloud Run Job ともに第一級対応。コールドスタートも Go に次いで軽量。

**クラウド適合性（AWS）**: Lambda・ECS Fargate ともにネイティブ対応。さらに **Amazon Bedrock** 経由で Claude を呼び出すことで、IAM 認証統合・VPC 閉域化・CloudWatch モニタリングという AWS 固有のメリットが得られる。外部 API への直接呼び出しを禁じるセキュリティポリシーを持つ小売企業の本番環境では、Bedrock 経由が実質的な必須要件になるケースがある。

#### Phase 2/3 適性：★★★☆☆

動的型付け言語であるため、コードベースが拡大すると型安全性と保守性が Java・Go・C# に劣る。業務ロジックが増えると構造が緩みやすく、責務分離が曖昧になりやすい。mypy・pydantic・pytest を全面適用する規律が必要であり、長期化した場合に構造の見直しを要するリスクがある。

---

### 4.2 Go

#### Phase 1 適性：★★★☆☆

API とバッチを軽量かつ高性能に構築しやすく、単一バイナリ配布・コンテナ実行との相性も良い。ただし、Claude API の公式 Go SDK が存在しないため HTTP クライアントの自作が必要であり、AI 解釈の試作速度は Python に劣る。データ加工・探索的処理は Python の方が楽である。

**クラウド適合性（GCP・AWS 共通）**: Cloud Run・Lambda ともにコールドスタートが最軽量クラスであり、高頻度のエンドポイントに最も向く。ただし AWS の Bedrock SDK（`boto3`）は Python 専用であり、Go から Bedrock を呼び出す場合は AWS SDK for Go v2 による独自実装が必要になる。

#### Phase 2/3 適性：★★★★☆

API とバッチの安定運用・並行処理・運用上の扱いやすさで優れる。**業務システム本体を Go で構築し、AI 解釈部分だけを Python ワーカーとして分離する構成**は、実務バランスが高い有力な選択肢である。

---

### 4.3 Java

#### Phase 1 適性：★★☆☆☆

PoC の初速は Python より遅い。JVM 起動コストによるコールドスタート問題は Cloud Functions 運用で深刻になりうる（GraalVM ネイティブイメージで緩和できるが設定コストが増す）。ボイラープレートが多く Phase 1 の PoC 速度を落とす。Claude API の公式 SDK も存在しない。

**クラウド適合性（GCP・AWS 共通）**: 技術的には対応しているが、他言語と比べて最適とは言えない。Spring Boot on Cloud Run / ECS Fargate は実績があるが重い。

#### Phase 2/3 適性：★★★★★

業務システムとしての成熟度が非常に高く、複雑なドメインモデルの整理・バッチ・トランザクション・監査・長期保守において安心感が高い。組織開発・チーム開発にも向く。**業務システム本体を Java で構築し、AI 解釈部分を Python ワーカーとして分離する構成**は自然で実績も多い。

---

### 4.4 C#

#### Phase 1 適性：★★★☆☆

ASP.NET Core は堅く、Java よりやや軽快な実装体験を持つ。ただし AI 解釈の試作速度は Python に劣り、Claude API の公式 SDK も存在しない。

**クラウド適合性**: .NET 8 以降は Linux コンテナファーストで設計されており、AWS ECS Fargate 上での稼働実績も十分にある。「C# は Azure 寄り」というのは .NET Core 以前の印象であり、現在は AWS + Linux コンテナでも問題なく動作する。詳細はセクション 6 で再評価する。

#### Phase 2/3 適性：★★★★☆

型安全性・保守性・Entity Framework Core による DB 操作の容易さが高い。Java 級の堅さと比較的軽快な実装体験を両立しやすく、業務システム本体としてかなり有力である。**C# 本体 + Python AI ワーカー**の分離構成は実務バランスが良い。

## 5. フェーズ別サマリー

### Phase 1（PoC）

| 言語 | 評価 | 理由 |
|---|---|---|
| Python | **最有力** | AI 解釈・データ処理・API を最速で立ち上げられる |
| Go | 有力 | API / バッチは強いが AI 試作は Python に劣る |
| C# | 有力だが初速は劣る | 業務システムとして堅いが AI 試作は Python に劣る |
| Java | 有力だが初速は劣る | 最も立ち上がりが重い |

**Phase 1 結論**: Python を第一候補とする。ただしこれは「PoC に限った最適解」であり、長期運用まで含めた唯一の正解を意味しない。

### Phase 2/3（本番運用・拡張）

| 言語 | 評価 | 理由 |
|---|---|---|
| Java | **非常に有力** | 長期運用・業務ドメイン・組織開発に強い |
| C# | **非常に有力** | 堅さと実装しやすさのバランスが良い |
| Go | 有力 | 軽量で運用しやすいが、ドメイン表現はやや素朴 |
| Python | 有力だが注意が必要 | 長期保守には型管理・規律の徹底が必要 |

## 6. 組み合わせ言語の優先度評価（AWS + Linux コンテナ前提）

Python が AI 解釈層として固定される前提で、業務システム本体の言語を評価する。評価環境は **AWS ECS Fargate（Linux コンテナ）** とする。

### 6.1 C# の AWS Linux コンテナ評価（再評価）

前版では「Azure 寄り」と記載したが、これは .NET Core 以前の印象であり、現在の .NET 8 以降では正確ではない。以下の通り再評価する。

**.NET 8 on Linux コンテナの現状**

.NET 8 は Linux ファーストで設計されており、`mcr.microsoft.com/dotnet/aspnet:8.0` の公式 Linux イメージが提供されている。AWS ECS Fargate 上での稼働実績も十分にある。AOT（Ahead-of-Time）コンパイルにより起動速度も改善されており、「C# は Windows 依存」というのは過去の認識である。

**AWS との統合**

AWS SDK for .NET は成熟しており、RDS（PostgreSQL）・SQS・S3・CloudWatch との連携に不足はない。Lambda についても .NET 8 ランタイムが正式サポートされており、軽量な API や Webhook 受口に活用できる。

**Bedrock との関係**

業務本体が C# の場合、AI 解釈は Python ワーカーに切り出す構成を前提とするため、Bedrock / Claude API は Python 側が担う。C# から Bedrock を直接呼ぶ必要はない。

**結論**: AWS + Linux コンテナ環境における C# の適合性は、前版の評価より高い。業務本体の組み合わせ言語として Go と並ぶ有力候補である。

---

### 6.2 組み合わせ言語の優先度ランキング

以下は「Python AI ワーカー + 業務本体」構成における、業務本体言語の優先度である。

| 優先度 | 言語 | 理由 |
|---|---|---|
| **1位** | **C#** | .NET 8 の Linux コンテナ対応が成熟。型安全・ASP.NET Core の開発体験・Entity Framework Core による DB 操作が揃っており、AWS との統合も問題ない。Java と同等の堅さをより軽快に実現できる |
| **2位** | **Go** | Linux コンテナとの親和性が最も高く、バイナリが小さくコールドスタートが速い。AWS ECS Fargate との相性も最良。ただし複雑な業務ドメインの表現は C# / Java に劣る |
| **3位** | **Java** | 業務ドメインの表現力と長期保守性は最高水準。ただし JVM コンテナイメージが重く（300MB超）、C# / Go と比べてコンテナ効率がやや劣る。既存 Java 資産がある場合は逆転する |

**C# を1位とする主な根拠**

- .NET 8 on Linux の成熟度が Go / Java に並ぶ水準に達している
- 型安全性・保守性・開発体験のバランスが3言語中最も高い
- コンテナイメージサイズが Java より軽量（公式 runtime イメージ: .NET 8 ≒ 220MB、JRE ≒ 300MB超）
- ASP.NET Core の Minimal API により Go に近い軽量な API 構成も可能
- AWS SDK for .NET が成熟しており、SQS / RDS / CloudWatch との連携に不足がない

### 6.3 推奨構成パターン

### パターン A：Python 単独

**向いているケース**: Phase 1 PoC、少人数、AI 解釈の改善を最優先する場合

**注意点**: 責務分離・型管理・テスト規律の徹底が必須。長期化した場合に構造見直しが必要になりやすい

**AWS 構成例**

| 用途 | サービス |
|---|---|
| AI 解釈パイプライン | Python + `boto3`（Bedrock）on Lambda / ECS |
| REST API | FastAPI on ECS Fargate |
| スコア算出バッチ | Lambda + EventBridge Scheduler |
| DB | RDS PostgreSQL |
| キュー | SQS |

---

### パターン B：C#（推奨）本体 + Python AI ワーカー

**向いているケース**: Phase 2 以降の本番運用を見据えた構成。業務ロジックを堅く保ちつつ AI 解釈を独立して改善したい場合

**AWS 構成例**

| 役割 | 言語 | サービス |
|---|---|---|
| REST API / スコア算出 / 業務ロジック | C# / ASP.NET Core | ECS Fargate（Linux コンテナ） |
| AI 解釈パイプライン / TrustEvent 生成 | Python + `boto3` | ECS Fargate（Linux コンテナ）または Lambda |
| サービス間連携 | — | SQS（AI 解釈ジョブのキュー） |
| DB | — | RDS PostgreSQL |

**メリット**: 本体の型安全性・保守性を確保しつつ、AI 解釈部分だけ Python で高速改善できる。Linux コンテナとしてどちらも ECS Fargate に並べて運用できるため、インフラの統一性が高い

---

### パターン C：Go 本体 + Python AI ワーカー

**向いているケース**: コンテナ効率・起動速度・軽量運用を最優先する場合

**特徴**: コンテナイメージが最小（数十 MB）でコールドスタートが最速。ただし複雑な業務ドメインの表現は C# より素朴になる。スタッフタグ入力の高頻度エンドポイントや Webhook 受口には特に有効

---

### パターン D：Java 本体 + Python AI ワーカー

**向いているケース**: 既存 Java 資産との統合が必要、または組織として Java のガバナンスが確立している場合に限定して検討する

## 7. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Python の動的型付けによる保守性低下（パターン A） | Phase 2 以降のバグ増加 | mypy・pydantic の全面適用。型アノテーションをコーディング規約に明記 |
| パターン B のサービス間 I/F 設計コスト | Phase 1 の立ち上げ遅延 | Phase 1 は Python 単独で始め、Phase 2 で分離。最初から分割しない |
| クラウドを AWS とした場合の Bedrock 呼び出し抽象化 | 直接 API と Bedrock API の混在 | AI 解釈クライアントを抽象クラスで包み、呼び出し先を設定で切り替え可能にする |
| AI 解釈の誤分類（言語に依らない共通リスク） | 誤った改善施策 | confidence 閾値・人間レビューキュー・週次精度検証（設計書の規定通り） |

## 8. 推奨方針まとめ

| 優先事項 | 推奨 |
|---|---|
| PoC 最速（Phase 1） | **Python 単独（パターン A）** |
| 本番運用・保守性バランス重視 | **C# 本体 + Python AI ワーカー（パターン B）← 全体推奨** |
| コンテナ効率・軽量運用重視 | **Go 本体 + Python AI ワーカー（パターン C）** |
| 既存 Java 資産との統合 | **Java 本体 + Python AI ワーカー（パターン D）** |

---

## 9. 結論

subjective-trust-platform の言語選定は、以下のように整理するのが最も中立かつ実務的である。

**「Phase 1 は Python を第一候補とし、Phase 2 以降は構造分離を前提に再評価する」**

Python を採用する場合は、設計段階から AI 解釈部分を疎結合に設計しておくことで、将来のパターン B への移行コストを最小化できる。Python をそのまま継続するか、業務本体を別言語で再構成するかは、Phase 1 の実運用データとチーム状況をもとに判断することを推奨する。
