## 1. 設計方針

### 1.1 利用シーンと画面の対応

本部担当者の利用シーンは4つある。単一の画面ですべてを賄うのではなく、**シーンに応じて使う画面が自然に変わる**構造にする。

| 利用シーン | 主に使う画面 | 頻度 |
|---|---|---|
| 問題店舗の早期発見 | ① ヒートマップ | 週次 |
| 月次会議の資料 | ① ヒートマップ + ② 散布図 | 月次 |
| 改善施策の効果を追う | ③ テーマ分布 + ① ヒートマップの推移 | 隔週 |
| ベストプラクティスを見つける | ② 散布図 + ③ テーマ分布 | 月次 |

### 1.2 is_reliable=false の表示方針

データ蓄積中（`is_reliable=false`）の店舗は**グレーアウトして表示する**。

- 存在は見せる：導入状況の把握と、蓄積の進捗確認のため
- スコアは表示するが「参考値」バッジを付ける
- ランキングや比較の対象からは除外する（ソート時に最下段に固定）
- ツールチップで「あと○件でスコアが確定します」を表示する

### 1.3 売上軸の正規化

散布図のX軸は**前週比（%）**で統一する。

- 基準線：前週比100%（変化なし）を中央に置く
- 表示範囲：70%〜130%（外れ値は端点にクリップし、ツールチップで実値を表示）
- 理由：店舗規模の差を吸収し、「売上が伸びているか落ちているか」の文脈で信頼スコアを見られる

## 2. 全体レイアウト

```
┌─────────────────────────────────────────────────────────┐
│  HEADER: 期間セレクタ（週/月/3ヶ月）  絞り込み（地域/フォーマット）  │
├─────────────────────────────────────────────────────────┤
│  SUMMARY BAR: 全店舗の総合スコア平均 / 前週比 / 要注意店舗数    │
├──────────────┬──────────────────────────────────────────┤
│              │                                          │
│  LEFT NAV    │  MAIN CONTENT（タブ切替）                │
│  ① ヒートマップ │  ① / ② / ③ の各画面                    │
│  ② 散布図    │                                          │
│  ③ テーマ分布 │                                          │
│              │                                          │
└──────────────┴──────────────────────────────────────────┘
```

SUMMARY BARは全画面共通で常時表示する。「要注意店舗数」は信頼スコアが閾値以下（初期値：総合スコア40以下）の`is_reliable=true`店舗数を示す。

## 3. 画面① ヒートマップ（店舗×5次元スコア）

### 3.1 目的

- 問題店舗の早期発見
- 月次会議での全店舗一覧の提示
- 特定次元の横断的な問題検知

### 3.2 レイアウト

```
┌──────────────────────────────────────────────────────────────┐
│ [ソート: 総合▼] [表示: スコア / 前週差 / 両方]  [エクスポート]  │
├──────────┬──────┬──────┬──────┬──────┬──────┬──────┬────────┤
│ 店舗名   │ 総合 │ 商品 │ 接客 │ 提案 │ 運営 │ 物語 │ 更新日 │
├──────────┼──────┼──────┼──────┼──────┼──────┼──────┼────────┤
│ 店舗A    │  72  │  78  │  65  │  70  │  80  │  68  │ 03/21  │
│ 店舗B    │  68  │  72  │  60  │  55  │  74  │  70  │ 03/21  │
│ ░店舗C░  │ (54) │ (60) │ (48) │ (55) │ (58) │ (50) │ 03/18  │
│  ↑ is_reliable=false: グレーアウト・参考値バッジ              │
└──────────┴──────┴──────┴──────┴──────┴──────┴──────┴────────┘
```

### 3.3 セルの色定義

| スコア範囲 | 色 | 意味 |
|---|---|---|
| 80〜100 | 濃い緑 | 良好 |
| 65〜79 | 薄い緑 | 概ね良好 |
| 50〜64 | 黄 | 要注意 |
| 35〜49 | 薄い赤 | 警戒 |
| 0〜34 | 濃い赤 | 要対応 |
| is_reliable=false | グレー | 参考値 |

### 3.4 表示モード切替

- **スコア表示**：現在の絶対値（デフォルト）
- **前週差表示**：+5 / −3 のような差分。色は増減方向（緑/赤）
- **両方表示**：「72 (+3)」のように併記。月次会議資料向け

### 3.5 インタラクション

**セルのホバー**：ツールチップで「前週: 68 → 今週: 72 (+4）/ 算出イベント数: 32件」を表示。

**行のクリック**：その店舗の12週推移グラフをドロワーで右側に展開する（画面遷移しない）。ドロワーには5次元の推移折れ線グラフと、直近のTrustEventサマリー上位3件を表示する。

**列ヘッダのクリック**：その次元でソート。問題次元を横断的に並べ替えて見られる。

**is_reliable=falseセルのホバー**：「データ蓄積中：あと約○件でスコアが確定します（現在○件/週あたり20件必要）」を表示。

### 3.6 エクスポート

「エクスポート」ボタンでCSV出力。月次会議資料への貼り付けを想定。出力内容：店舗名・5次元スコア・前週差・is_reliable・更新日。

## 4. 画面② 散布図（売上前週比 × 信頼スコア）

### 4.1 目的

- 「売上は伸びているが信頼が落ちている」象限の発見（短期売上優先による信頼毀損の検知）
- ベストプラクティス店舗の特定（売上も信頼も伸びている象限）
- 改善施策の効果追跡（施策実施前後のプロット変化）

### 4.2 レイアウト

```
                        信頼スコア（Y軸）
                        100
                         │
            要注意        │        優良
     売上↑・信頼↓         │   売上↑・信頼↑
                         │
  ──────────────────────○──────── 売上前週比（X軸）
        70%      基準(100%)      130%
                         │
            低迷          │        改善余地
     売上↓・信頼↓         │   売上↓・信頼↑
                         │
                          0
```

**4象限の意味と優先度**

| 象限 | 売上 | 信頼 | 解釈 | 優先度 |
|---|---|---|---|---|
| 右上 | ↑ | ↑ | 優良。ベストプラクティス抽出対象 | 参照 |
| 左上 | ↓ | ↑ | 信頼は高い。売上施策の検討余地 | 中 |
| 右下 | ↑ | ↓ | **最優先対応**。信頼毀損が進行中 | 🔴 高 |
| 左下 | ↓ | ↓ | 複合的な課題。要精査 | 高 |

### 4.3 プロットの仕様

- **点のサイズ**：来店客数（大きいほど来客数が多い）
- **点の色**：5次元のうち最も低いスコアの次元を色分け（商品/接客/提案/運営/物語）
- **is_reliable=false**：点をグレーかつ破線の輪郭で表示。4象限の判定対象からは除外

### 4.4 インタラクション

**プロットのホバー**：ツールチップで「店舗名 / 売上前週比: +8% / 総合信頼スコア: 58（前週: 63）/ 最低次元: 接客信頼(48)」を表示。

**プロットのクリック**：ヒートマップ画面のドロワーと同じ12週推移グラフを展開する。

**次元フィルタ**：Y軸のスコアを「総合」から特定次元（接客信頼など）に切替可能。月次会議で「接客信頼だけで見ると」という議論に対応する。

**施策マーカー**：Phase 3以降の施策管理機能と連携し、施策実施日をX軸上にマーカーとして重ねて表示する（Phase 2では非表示・設計だけ予約）。

### 4.5 X軸の外れ値処理

前週比が70%未満または130%超の店舗は端点（70% / 130%）にクリップし、ツールチップで「実際の前週比: 145%」のように実値を表示する。クリップされた点には▲マークを付ける。

## 5. 画面③ テーマ分布（店舗×テーマタグ）

### 5.1 目的

- どの店舗でどの不満テーマが多いかを俯瞰する
- 改善施策の効果追跡（「押し売り感」タグが減ったか）
- ベストプラクティスの特定（特定テーマがゼロの店舗の接客パターン確認）

### 5.2 レイアウト

```
┌──────────────────────────────────────────────────────────────┐
│ [感情: ネガティブ▼] [次元: すべて▼] [期間: 直近4週▼]          │
├──────────┬────────┬──────┬────────┬──────┬────────┬──────────┤
│ テーマ   │ 店舗A  │店舗B │ 店舗C  │店舗D │ 店舗E  │ 全店計  │
├──────────┼────────┼──────┼────────┼──────┼────────┼──────────┤
│ 押し売り感│  ●●●  │  ●  │  ●●   │  ─  │  ●●●● │    9    │
│ 説明不足  │  ●●   │ ●●● │  ─    │  ●  │  ─    │    6    │
│ 欠品対応  │  ─    │  ─  │ ●●●●  │  ─  │  ─    │    4    │
│ 丁寧さ ✓ │  ●●●  │  ─  │  ─    │ ●●● │  ●●   │    8    │
└──────────┴────────┴──────┴────────┴──────┴────────┴──────────┘
  ●の大きさ = 件数  ─ = 0件  ✓ = ポジティブテーマ
```

### 5.3 表示仕様

- **バブルサイズ**：件数に比例（最大サイズ = 当該期間の最大件数）
- **バブルの色**：ネガティブ=赤系、ポジティブ=緑系
- **ゼロ件**：「─」で表示（空白にしない。意図的にゼロであることを見せる）
- **テーマの並び順**：デフォルトは「全店計の多い順」。列ヘッダクリックで特定店舗の多い順にソート可能
- **is_reliable=false店舗**：列をグレーアウト

### 5.4 フィルタ仕様

- **感情フィルタ**：ネガティブのみ（デフォルト）/ ポジティブのみ / 両方
- **次元フィルタ**：接客信頼のみ、など5次元で絞り込み
- **期間フィルタ**：直近4週（デフォルト）/ 直近8週 / 直近12週

### 5.5 インタラクション

**バブルのホバー**：ツールチップで「店舗A / 押し売り感 / 4件（前同期間: 2件）」を表示。

**バブルのクリック**：対象テーマ×店舗のTrustEventサマリーをモーダルで表示する。表示内容：`interpretation`フィールドの要約（最大5件）、発生した接客日時、confidence値。これが「なぜそのテーマが多いか」の根拠確認に使う。

**テーマ行のホバー**：そのテーマが多い店舗と少ない店舗のパターン差が浮き彫りになる。ベストプラクティス発見の主要経路。

## 6. 共通仕様

### 6.1 期間セレクタ

全画面共通のヘッダに配置する。

| 選択肢 | 用途 |
|---|---|
| 直近1週 | 週次モニタリング |
| 直近4週 | 月次会議資料（デフォルト） |
| 直近12週 | 施策効果の追跡・Quarter振り返り |

### 6.2 絞り込みフィルタ

| フィルタ | 選択肢 |
|---|---|
| 地域 | 関東 / 関西 / 東海 / 全国 |
| フォーマット | 直営のみ（Phase 2スコープ）|
| is_reliable | 確定済みのみ / すべて（デフォルト） |

### 6.3 月次会議モード

ヘッダに「会議モード」トグルを設ける。ONにすると以下が変わる。

- ヒートマップ：「両方表示」（スコア + 前週差）に切替
- グレーアウト店舗：完全非表示に切替（会議資料として煩雑さを除く）
- エクスポートボタン：全3画面分のデータを一括CSV出力

### 6.4 ローディング状態

スコアデータの取得中は、セルをスケルトン表示（グレーのパルスアニメーション）にする。部分的に描画されたヒートマップが一瞬表示される状態を避ける。

## 7. API設計（フロントエンド向け）

本部分析画面が必要とするエンドポイントを定義する。Trust API（Phase 3）の先行設計として位置づける。

```
# ① ヒートマップ用
GET /api/v1/stores/scores
  ?snapshot_date=2026-03-21
  &include_unreliable=true

Response:
[{
  "store_id": "...",
  "store_name": "店舗A",
  "is_reliable": true,
  "scores": {
    "overall": 72, "product": 78, "service": 65,
    "proposal": 70, "operation": 80, "story": 68
  },
  "prev_week_diff": {
    "overall": +4, "product": +2, ...
  },
  "snapshot_date": "2026-03-21"
}]

# ② 散布図用
GET /api/v1/stores/scatter
  ?trust_dimension=overall
  &period_weeks=4

Response:
[{
  "store_id": "...",
  "store_name": "店舗A",
  "trust_score": 72,
  "sales_wow_pct": 108.3,   // 前週比%
  "visit_count": 420,        // バブルサイズ用
  "lowest_dimension": "service",
  "is_reliable": true
}]

# ③ テーマ分布用
GET /api/v1/stores/themes
  ?sentiment=negative
  &period_weeks=4
  &trust_dimension=all

Response:
{
  "themes": ["押し売り感", "説明不足", ...],
  "stores": ["店舗A", "店舗B", ...],
  "matrix": {
    "押し売り感": {"店舗A": 3, "店舗B": 1, ...},
    ...
  }
}

# ドロワー用（①②共通）
GET /api/v1/stores/{store_id}/trend
  ?weeks=12

# テーマ詳細モーダル用（③）
GET /api/v1/stores/{store_id}/trust-events
  ?theme_tag=押し売り感
  &period_weeks=4
```

## 8. 実装優先順序

Phase 2の実装着手時に、以下の順序で進める。

| 順序 | 対象 | 理由 |
|---|---|---|
| 1 | APIエンドポイント（7節） | フロントと並行開発するための先行実装 |
| 2 | ① ヒートマップ | 最も利用頻度が高く、全利用シーンに関与する |
| 3 | ③ テーマ分布 | ヒートマップのドリルダウン先として早期に必要 |
| 4 | ② 散布図 | 月次会議・ベストプラクティス発見で使うが、①③より補助的 |
| 5 | 月次会議モード | ①〜③が揃ってから追加 |
