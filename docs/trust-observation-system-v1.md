## 1. 目的と前提

### 1.1 解決する課題

実店舗では、ブランド信頼の形成と毀損が日常的に起きている。しかし売上データと日報だけではそれを捉えられない。

典型的な見落としは以下のとおり。

- 接客は行われたが不快感により離脱した
- 商品は良いが説明不足で購入に至らない
- 欠品時の対応が悪くブランド全体への印象が下がる
- 短期売上は出ているが再来店意向が下がっている
- 接客は丁寧で今回は買わなくても次回は来たいと思った——が、それが記録されない

これらはすべて **顧客の主観** の領域で起きている。購買の有無やPOSの数値には現れない、「どう感じたか」「なぜそう受け取ったか」が信頼の実体である。

本システムは、この主観を構造的に記録・解釈し、改善ループを回すための基盤である。

### 1.2 中核概念：信頼は主観の業務表現である

ブランド運営で使う「信頼」は抽象語に見えるが、実態は顧客主観の積み重ねである。

- この店なら安心して相談できる
- 自分に合わないものを無理に勧めてこない
- 商品説明に納得できる
- 欠品時も誠実に対応してくれる
- このブランドは世界観がぶれていない

したがって信頼観測とは、顧客主観のうち特に **安心・納得・違和感・不快感・期待一致** を観測対象とする仕組みである。予測や売上分析ではなく、「なぜそう感じたか」を解釈し、改善につなげることが本質的な目的となる。

### 1.3 位置づけと制約

本システムは **店舗スタッフの監視ツールではなく、ブランド信頼の改善支援基盤** である。

設計上の制約として以下を置く。

- 個人査定の直接材料にしない。評価対象は「信頼接点」であり「個人の優劣」ではない
- 現場の入力負荷を1接客あたり10秒以内に抑える
- AI出力はすべて仮説提示であり、確定判断として扱わない
- 短期売上最適化と信頼KPIを明確に分離する

### 1.4 スコープ

Phase 1（本設計の対象）は、直営店舗5〜10店規模での最小構成導入とする。卸先店舗・ポップアップは対象外。

## 2. 信頼の定義と観測モデル

### 2.1 5つの信頼次元

| 次元 | 定義 | 主な観測源 |
|---|---|---|
| 商品信頼 | 期待した品質・機能・価格納得感があるか | 返品理由、品質レビュー、素材・耐久性への不満 |
| 接客信頼 | 安心して相談できるか、不快な思いをしないか | 接客後アンケート、接客後離脱率、クレーム記述 |
| 提案信頼 | 自分に合った提案がされているか | 提案後購買率、提案後返品率、欠品時代替提案評価 |
| 運営信頼 | 案内・在庫・受取・価格表示に齟齬がないか | 在庫確認正確性、レジ待ち時間、EC-店舗間の案内差異 |
| 物語信頼 | ブランドらしさと一貫性が保たれているか | 世界観レビュー、値引き施策後の反応、空間-接客-商品訴求の整合性 |

### 2.2 顧客主観の三層モデル：Trait / State / Meta

信頼の形成・毀損は、顧客側の文脈によって意味が変わる。同じ接客でも、顧客の価値観・その日の状況・過去の体験によって受け取り方は異なる。この文脈を構造化するために、顧客主観を以下の三層で捉える。

**Trait（長期的な価値観・選好）**

顧客が継続的に重視する軸。来店をまたいで安定しており、大きくは変わらない。

例：品質重視で安さには関心が薄い、押し売りを強く嫌う、丁寧に相談したい、世界観やブランドらしさを重視する、長く使えるものを求める。

**State（来店時の短期的な状態）**

その来店時点の目的・感情・制約。来店ごとに変わりうる。

例：今日は下見だけ、ギフトなので失敗したくない、急いでいる、予算が厳しい、たまたま立ち寄った。

**Meta（違和感・修正の履歴）**

過去の体験から蓄積された違和感や期待の修正。信頼毀損の「伏線」として機能する。

例：前回の接客が合わなかった、欠品時の対応で不信感が残っている、価格説明に納得できなかった経験がある、想定と違う提案を受けた。

この三層は、AIが自由記述や接客メモを解釈する際の分析枠組みとして機能する。たとえば「押し売りされた」というコメントに対して、それが顧客のTrait（もともと押し売りを嫌う傾向）に起因するのか、State（急いでいたので提案が過剰に感じた）に起因するのか、Meta（前回も似た経験をした二度目）に起因するのかで、改善施策の方向性が変わる。

Phase 1ではTrait / State / Metaの完全な推定は行わない。ただし、接客タグの「来店目的」がStateに、アンケートの自由記述や接客メモがMeta手がかりに対応するよう入力設計を行い、将来的にAIが三層を推定できるデータ基盤を整える。

### 2.3 信頼イベントモデル

信頼の変動は「信頼イベント」として記録する。信頼イベントとは、5次元のいずれかにポジティブまたはネガティブな影響を与えた出来事の構造化レコードである。

信頼イベントは以下の3経路で生成される。

**手動生成**: スタッフの接客タグ入力、顧客ミニアンケート回答から、ルールベースで信頼イベントを生成する。例えば「接客結果=欠品離脱」かつ「代替提案=なし」の組み合わせは、提案信頼のネガティブイベントとして自動登録する。

**自動生成**: POS・会員データから、返品発生、再来店間隔の異常延長、特定カテゴリの購買率急落などをトリガーにイベントを生成する。

**AI生成**: 自由記述コメント、外部レビュー、問い合わせログをLLMで解釈し、信頼次元・感情・論点を分類したうえでイベント化する。ここでの「解釈」とは、単にキーワードを検出するのではなく、「なぜそう感じたか」の構造（どの信頼次元が、どのような文脈で、どの程度毀損・形成されたか）を推定する処理を指す。

### 2.4 信頼スコア算出ロジック

各信頼次元のスコアは、当該次元に紐づく信頼イベントの加重集計で算出する。スコアは0〜100のスケールで表現し、週次スナップショットとして保存する。

**算出式（次元ごと）:**

```
dimension_score = base_score 
  + Σ(positive_event_weight × recency_decay) 
  - Σ(negative_event_weight × severity × recency_decay)
```

- `base_score`: 初期値50。データ不足時はこの値に回帰する
- `positive_event_weight / negative_event_weight`: イベント種別ごとに定義（後述の重みテーブルで管理）
- `severity`: 1〜3の3段階。AI分類または手動設定
- `recency_decay`: 直近4週は1.0、5〜8週は0.7、9〜12週は0.4、それ以前は0.1

**重みテーブル（例: 接客信頼）:**

| イベント種別 | 方向 | 基本重み | 備考 |
|---|---|---|---|
| アンケート高評価（4-5点） | + | 2.0 | |
| アンケート低評価（1-2点） | - | 3.0 | 毀損は形成より影響大 |
| 接客後購買 | + | 1.0 | |
| 接客後離脱 | - | 1.5 | |
| 「押し売り」分類コメント | - | 4.0 | 重度の信頼毀損 |
| 「丁寧」分類コメント | + | 2.5 | |
| クレーム（接客起因） | - | 5.0 | |

重みテーブルはPhase 1導入後の実データを用いて四半期ごとに見直す。初期値は定性判断で設定し、再来店率・NPS等の外部指標との相関分析で調整していく。

**総合スコア:**

5次元の加重平均とする。デフォルトの重みは均等（各0.2）とし、ブランド戦略に応じて調整可能にする。ただし総合スコアは補助指標であり、意思決定は次元別スコアで行う。

### 2.5 コールドスタート対応

導入直後はデータ量が不足するため、以下の段階的運用を行う。

**〜4週目（蓄積期）**: スコア算出は行わない。イベント収集とAI分類精度の検証に集中する。ダッシュボードにはイベント件数と分類結果の一覧のみ表示する。

**5〜12週目（試行期）**: スコア算出を開始するが「参考値」と明示する。base_scoreへの回帰が強く効く期間であることをUIで注記する。

**13週目〜（本運用）**: スコアの信頼区間が十分に狭まった段階で、前週比・アラートを有効化する。十分かどうかの判定は、週あたりイベント数が次元ごとに20件以上を閾値とする。

## 3. システムアーキテクチャ

### 3.1 全体構成

```
[入力層]                    [解釈層]                    [出力層]
                                                        
POS/会員DB ──┐              ┌─ イベント生成 ──┐          ┌─ 店舗ダッシュボード
接客タグ入力 ─┤              │  エンジン       │          ├─ 本部分析画面
ミニアンケート┼─→ データ収集 ─┼─ AI解釈      ──┼─→ 信頼DB ┼─ 週次レポート
問い合わせDB ─┤   パイプライン│  パイプライン   │          ├─ アラート通知
外部レビュー ─┤              │                 │          └─ 改善提案レポート
店舗日報 ────┘              └─ スコア算出 ───┘          
```

処理層を「解釈層」と呼ぶ。これは意図的な命名であり、本システムにおけるAIの役割が「予測」ではなく「解釈」——顧客がどう感じ、なぜそう受け取ったかを構造的に読み解くこと——にあることを設計レベルで明示するためである。

### 3.2 技術スタック（Phase 1想定）

**データ収集・蓄積**: Cloud SQL（PostgreSQL）。Phase 1の規模ではRDBで十分。将来的なイベント量増大に備え、TrustEventテーブルはパーティショニング設計を入れておく。

**AI解釈パイプライン**: Claude API（Sonnet）を利用。自由記述テキストを入力とし、信頼次元・感情・論点タグ・重要度・主観文脈（Trait/State/Metaの手がかり）をJSON形式で返却するプロンプトを定義する。バッチ処理で日次実行し、結果をTrustEventテーブルに書き込む。

**スコア算出**: 日次バッチジョブ（Cloud Functions or Cloud Run Job）。イベントテーブルを集計し、TrustScoreSnapshotを更新する。

**ダッシュボード**: Phase 1ではLooker Studio + カスタムフロントエンド（Next.js）のハイブリッド構成。Looker Studioで本部向けの探索的分析を、カスタムフロントエンドで店舗向けの日常利用画面を提供する。

**接客タグ入力**: モバイルWebアプリ（PWA）。スタッフ個人端末または店舗タブレットから入力。オフライン入力対応のため、ローカルキューイング+オンライン時同期の設計とする。

### 3.3 AI解釈プロンプト設計方針

自由記述の解釈には以下の出力スキーマを定義する。

```json
{
  "trust_dimension": "service | product | proposal | operation | story",
  "sentiment": "positive | negative | neutral",
  "severity": 1 | 2 | 3,
  "theme_tags": ["押し売り感", "説明不足", ...],
  "summary": "1文の要約",
  "interpretation": "なぜそう感じたと推定されるかの1文解釈",
  "subjective_hints": {
    "trait_signal": "品質重視の傾向" | null,
    "state_signal": "ギフト選びで慎重になっていた" | null,
    "meta_signal": "過去の欠品体験への不満が再燃" | null
  },
  "confidence": 0.0〜1.0
}
```

`interpretation` フィールドは、単なる分類結果に「なぜ」の仮説を付与するためのものである。例えば「押し売り感」という分類だけでは改善方向が見えないが、「ギフト選びで慎重になっていた顧客に対し、複数商品の提案が過剰に映った」という解釈があれば、改善のとりかかりになる。

`subjective_hints` はTrait/State/Metaの手がかりを抽出するフィールドである。Phase 1ではAI出力の参考情報として蓄積し、Phase 2以降で顧客ごとの主観プロファイル構築に活用する。

confidence が 0.6 未満の場合は「要確認」フラグを立て、人間レビューキューに入れる。この閾値はPhase 1運用中に精度検証のうえ調整する。

分類精度の目標は、信頼次元の正答率85%以上、感情分類の正答率90%以上とする。週次で100件のサンプリング検証を行い、精度が目標を下回った場合はプロンプトを改訂する。

## 5. 入力設計

### 5.1 接客タグ入力（スタッフ向け）

接客終了後、スタッフがモバイルWebで以下を入力する。目標入力時間は10秒以内。

**必須項目（タップ選択のみ）:**
- 来店目的: 購入目的 / 下見 / 比較検討 / ギフト / 修理・問合せ
- 接客結果: 購入 / 検討継続 / 離脱 / 欠品離脱

来店目的は顧客のState（その来店時の短期的状態）を捉える設計意図を持つ。

**条件付き表示:**
- 欠品離脱の場合のみ: 代替提案した / 取り寄せ案内した / いずれもなし
- 離脱の場合のみ: 不安点（複数選択可）: 価格 / サイズ仕様 / 素材品質 / 利用シーン / 他社比較

**任意項目:**
- 次回来店見込み: 高 / 普通 / 低
- 特記メモ: 自由記述（1行程度）

特記メモは、顧客のMeta（過去体験への言及や違和感の表出）を拾う主要ソースとなる。「前回も同じ商品が欠品だった」「以前別店舗で嫌な思いをしたと言っていた」等の情報が記録されることで、AIが信頼毀損の構造を解釈する手がかりになる。

UIはスマートフォンの片手操作で完結する設計とし、最小2タップ（来店目的+接客結果）で送信可能とする。

### 5.2 顧客ミニアンケート

来店翌日にLINEまたはメールで配信する。3問+任意コメント。所要時間30秒以内を目指す。

1. 本日の接客は安心して相談できましたか（5段階）
2. 欲しい情報は十分得られましたか（5段階）
3. またこの店舗で相談したいですか（5段階）
4. ご意見があればお聞かせください（任意自由記述）

Q1〜Q3は信頼の定量観測、Q4は主観の定性観測として機能する。Q4の自由記述がAI解釈パイプラインの主要入力となり、Trait/State/Metaの手がかり抽出元になる。

回答率目標: Phase 1で15%、Phase 2で25%。回答後にポイント付与（10〜50pt程度）を検討する。

### 5.3 自動連携

| データソース | 連携方式 | 頻度 | 取得項目 |
|---|---|---|---|
| POS | DB連携（バッチ） | 日次 | 購入金額、商品カテゴリ、値引額、返品 |
| 会員DB | API連携 | 日次 | 会員ランク、来店履歴、購買履歴 |
| 問い合わせDB | API連携 | 日次 | 分類、対応時間、解決状況 |
| Googleビジネスプロフィール | API連携 | 日次 | 評価点、レビューテキスト、投稿日 |

## 6. 画面設計

### 6.1 店舗ダッシュボード（店長・店舗責任者向け）

**利用頻度**: 日次〜週次

**画面構成:**

上段 — KPIカード4枚を横並び表示。総合信頼スコア（前週比付き）、接客後離脱率、再来店意向率（アンケートQ3の平均）、欠品起因不満件数。

中段 — 5次元別スコアのレーダーチャートと、過去12週の推移折れ線グラフを左右に配置。レーダーチャートは前週と今週の重ね表示。折れ線グラフは任意の次元をトグルで表示切替可能。

下段 — 3カラム構成。左に「今週のAI要約」（3〜5文の自然言語要約）、中央に「注目アラート」（最大5件）、右に「改善提案」（AI生成の施策案、最大3件）。

AI要約は単なる数値まとめではなく、「なぜ」の解釈を含む形式で出力する。例：「欠品時の代替提案率が低下しており、特にギフト目的の来店客で提案信頼が下がっています。ギフト購入者は代替案への期待が高い傾向があるため、重点的な対応が推奨されます。」

**コールドスタート期の表示**: スコアが未確定の期間は、レーダーチャートと推移グラフの代わりに「データ蓄積中」の進捗バーとイベント件数カウンターを表示する。

### 6.2 本部分析画面

**利用頻度**: 週次〜月次

**主要ビュー:**

店舗比較ビュー — 店舗を行、5次元スコアを列としたヒートマップ。色でスコア水準を直感的に把握できるようにする。行はソート・フィルタ可能。

散布図ビュー — X軸に売上、Y軸に信頼スコア（次元選択可能）をプロットし、「売上は高いが信頼が低い」象限の店舗を視覚的に発見する。この象限は、短期売上が出ているが主観レベルで信頼毀損が進行している状態を示し、最も優先的に改善すべき対象となる。

テーマ分布ビュー — 店舗×テーマタグのバブルチャート。どの店舗でどの不満テーマが多いかを俯瞰する。

### 6.3 接客改善レポート（週次自動生成）

毎週月曜朝にSlackまたはメールで店長に配信する。内容は以下。

- 今週増加した不満テーマ上位3件と具体例（匿名化済み）
- 高評価接客の共通パターン（AI抽出）——何をしたかだけでなく、どのような顧客文脈（State）で有効だったかを含む
- 欠品対応の改善余地（代替提案実施率の推移）
- AI生成の改善アクション提案（具体的な行動レベルで3件以内）

## 7. アラート設計

### 7.1 アラート条件と閾値

| アラート種別 | 条件 | 閾値（Phase 1初期値） | 確認観点 |
|---|---|---|---|
| 接客後離脱率上昇 | 週次離脱率が前4週平均の1.5倍超 | 前4週平均×1.5 | 初回声かけタイミング、高単価商品説明内容 |
| 押し売り感タグ急増 | 週あたり件数が前4週平均の2倍超 | 前4週平均×2.0 | 該当接客パターン、キャンペーン圧力 |
| 欠品不満継続 | 3週連続で欠品起因ネガティブイベントが増加 | 3週連続増加 | 代替提案実施率、取り寄せ案内率 |
| 再来店意向低下 | アンケートQ3平均が2週連続低下 | 2週連続0.3pt以上低下 | 直近の店舗運営変更、スタッフ体制変更 |
| 外部レビュー低評価連続 | 1週間に星2以下が3件以上 | 3件/週 | レビュー内容の信頼次元分類 |
| 返品理由悪化 | 特定カテゴリの品質起因返品率が閾値超 | カテゴリ別返品率5%超 | 商品ロット、説明内容の乖離 |

### 7.2 アラート出力形式

アラートは「異常検知」と「確認すべき観点」をセットで出す。単なる数値通知にしない。

例:

> **欠品不満が3週連続で増加しています**
> 
> 提案信頼スコア: 62→58→54（3週推移）
> 
> AI解釈: ギフト目的の来店客で「代替提案なし」のイベントが集中しています。ギフト購入者は他のState（下見、比較検討）と比べて代替案への期待が高く、提案がないこと自体が強い不信につながる傾向があります。
> 
> 確認してください: 代替提案実施率（現在32%）、取り寄せ案内率（現在45%）
> 
> 推奨アクション: 欠品時の代替提案トークを標準化し、特にギフト来店客への対応を優先して共有する

### 7.3 閾値の調整

Phase 1の閾値は仮設定であり、3ヶ月運用後にアラート発報頻度と実際の改善行動率を検証して調整する。目標は週あたり1〜3件のアラートに収まる水準。多すぎるとノイズ化し無視されるため。

## 8. プライバシー・データガバナンス

### 8.1 顧客データ

- アンケート回答時にデータ利用目的と範囲を明示し、同意を取得する。同意ステータスはCustomerテーブルで管理する
- 自由記述コメントは店舗ダッシュボードでは匿名表示する。個人特定可能な情報（氏名・電話番号等）がコメントに含まれる場合はAI前処理でマスキングする
- 生データの保持期間は2年間とし、それ以降は集計済みスコアのみ保持する
- 外部レビューは公開情報として取り扱うが、個人を特定する二次利用は行わない
- Phase 2以降のSubjectiveProfile（Trait要約等）は、個人情報保護法における「個人関連情報」に該当しうるため、構築・利用にあたっては法務確認を必須とする

### 8.2 スタッフデータ

- スタッフ別の集計はPhase 1では行わない。Phase 2以降で検討する場合も、本人へのフィードバック用途に限定し、人事評価の直接材料としない運用ルールを明文化する
- 接客タグ入力は「店舗改善のためのデータ提供」として説明し、入力率は評価しても入力内容で個人を不利に扱わない

### 8.3 AI処理

- Claude APIへの送信データは自由記述テキストのみとし、顧客の個人識別情報を含めない
- AI分類・解釈結果は人間レビュー可能な状態を維持する。confidence < 0.6 の分類は自動でレビューキューに入る
- AI解釈プロンプトおよび重みテーブルの変更はバージョン管理し、変更履歴を残す
- AIの解釈結果（特にTrait/State/Meta推定）はあくまで仮説であることをUI上で明示し、確定事実として扱わない設計とする

## 9. ユースケース

### 9.1 店長が週次で店舗状態を把握する

月曜朝、店長は週次レポート（メール/Slack）で概況を確認する。「接客後離脱率が先週より上昇」「説明不足タグが増加」等の論点を把握する。レポートにはAI解釈が添えられており、「比較検討目的の来店客で離脱が増えている。他社比較への不安タグも増加しており、競合情報を踏まえた説明が不足している可能性がある」と書かれている。店長はダッシュボードで詳細を確認し、今週の朝礼で競合比較の説明ポイントを共有する。翌週のスコア変化で効果を確認する。

### 9.2 本部が店舗間差分を発見する

月次会議の前に、本部担当者が散布図ビューを開く。売上上位だが接客信頼が低い店舗Aを発見する。テーマ分布ビューで確認すると「押し売り感」タグが突出している。TrustEventの解釈フィールドを確認すると「下見目的の来店客に対して積極的なクロージングが行われている」パターンが多い。来店客のState（下見）と接客アプローチのミスマッチが信頼毀損の構造であると把握し、店舗Aの店長と改善計画を策定する。

### 9.3 商品部門が期待と実物のギャップを検知する

商品信頼スコアが特定カテゴリで低下傾向にあることを本部分析画面で発見する。TrustEventのテーマタグを確認すると「思ったより安っぽい」「EC写真と違う」が多い。AI解釈は「EC写真で形成された品質期待と、実物を手に取った際の質感のギャップが不信につながっている」と推定している。商品ページの写真・説明と実物の乖離を調査し、商品説明の改訂または店頭POPの追加で対応する。

## 12. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| スタッフが入力を形骸化する | データ品質低下→スコア信頼性喪失 | 入力を極限まで簡素化。入力率を運用KPIとして追跡。入力の意義を定期的に現場に伝える |
| AIの誤分類・誤解釈が改善施策を歪める | 誤った方向への改善→信頼毀損 | confidence閾値による人間レビュー。週次の精度検証。誤分類フィードバック導線。解釈結果をUIで「仮説」と明示 |
| スコアが個人評価に転用される | 現場の不信→入力拒否・虚偽入力 | Phase 1ではスタッフ別集計を行わない。運用ルールの明文化。導入時の趣旨説明の徹底 |
| アンケート回答率が低すぎる | Feedback由来の信頼イベント不足 | 回答インセンティブ設計。配信チャネル・タイミングの最適化。他データソースでの補完 |
| アラートが多すぎてノイズ化 | 店長がアラートを無視するようになる | 閾値を週1〜3件に調整。3ヶ月ごとの閾値レビュー |
| AI解釈の「なぜ」が的外れでも説得力を持つ | もっともらしい誤解釈が無批判に採用される | 解釈にはconfidenceを併記。レポートに「この解釈は仮説です」を常時表示。店長確認プロセスを必須化 |

## 13. まとめ

ブランド再生の中核は、売上の回復そのものではなく、顧客がそのブランドに対して再び安心と期待を持てる状態を作ることである。信頼とは顧客主観の積み重ねであり、その観測は「何が起きたか」だけでなく「なぜそう感じたか」の解釈まで踏み込む必要がある。

実店舗はその信頼が最も濃く形成され、同時に最も毀損されやすい場でもある。

本システムは、実店舗における信頼の形成・毀損を5つの次元で構造的に観測し、顧客主観の三層モデル（Trait/State/Meta）を手がかりにAIが解釈・要約・兆候検知を行い、改善ループを回すための運用基盤である。設計にあたっては、現場負荷の最小化、監視への転化の防止、AI解釈への過度な依存の回避、プライバシーの保護を制約として組み込んだ。

本設計はPhase 1の最小構成導入を対象とし、5店舗での実証を経て段階的に拡張する。Phase 2以降では、蓄積された主観手がかりを活用した顧客理解の深化と、接客前のコンテキスト提供まで射程に入れる。
